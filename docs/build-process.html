Decision Support Bot

תהליך הבניה

1
Backend — לוגיקת המלצה נקודת קצה POST /api/recommend + חישוב 4 ציונים (אופרציונלי, דיוק, כלכלי, אסטרטגי) וחוקי החלטה לבחירת ארכיטקטורה.
2
Frontend — טופס שאלות 12 שאלות עם range sliders (1–5), חישוב midpoint, שליחה ל-API והצגת ResultCard + BonusAlerts.
3
Hebrew Localization תרגום מלא לעברית, RTL layout, כל טקסטי הממשק, השאלות וההמלצות.
4
ChatBot UI המרת הטופס לחוויית צ'אט WhatsApp-style: שאלה אחת בכל פעם, slider dock תחתון, מסלולים דינמיים ו-redirect חכם.

Tech Stack ולוגיקת החלטה

Frontend React 18 + Vite + Lucide Icons
Backend Node.js + Express (stateless)
Deployment Render — שרת יחיד, API + static
בסיס נתונים אין — fully stateless per session
Operational Efficiency > Accuracy & Trust → Late Fusion
Accuracy & Trust ≥ Operational → Deep Feature Fusion
Economic avg ≥ 4 → + Knowledge Distillation
Strategic avg ≥ 4 → + Open-Source LLM

פרומפטים הבוט — שאלות הפתיחה

ברכת פתיחה (Greeting): "שלום! אני FraudGuard, כאן לעזור לך לבחור את הארכיטקטורה המתאימה לזיהוי הונאה. נתחיל בשאלה אחת חשובה שתכוון אותנו לכיוון הנכון עבורך."
שאלת מסלול (Opening): "היי! כדי שלא נבזבז זמן יקר, בוא נתחיל מההחלטה הכי גדולה... מהו ה'אני מאמין' המרכזי של החברה שלכם?"
מסלול 1 — יעילות אופרציונלית
"הכי חשוב לנו שהמערכת תעבוד מהר, תגיב בשבריר שנייה ולא תקרע לנו את הכיס"
מסלול 2 — דיוק ואמון
"הכי חשוב לנו דיוק מוחלט במעבדה, ושקיפות מלאה כדי שהמנהלים יבינו ויסמכו על המסקנות"

מסלול 1 — 7 שאלות (יעילות)

  • 1עד כמה דחוף לכם לתפוס את השקר בשידור חי תוך כדי השיחה (כמו זום), ולא בדיעבד?
  • 2עד כמה חשוב לכם שהמערכת תיתן התראה מיידית (בשבריר שנייה) ברגע שמזהה חוסר אמינות?
  • 3עד כמה חשוב לכם לקבל הסבר ברור במילים (ולא רק ציון יבש באחוזים)?
  • 4עד כמה מחמיר ומוגבל התקציב שלכם לרכישת שרתים יקרים וכרטיסי מסך (GPUs)?
  • 5עד כמה חשוב לכם לשמור על עלויות תחזוקה והרצת מודלים נמוכות ביומיום?
  • 6האם הראיונות שלכם מתבצעים בתנאי שטח קשים (רעשי רקע, אינטרנט חלש, וידאו מטושטש)?
  • 7עד כמה מאתגר הצורך שלכם לסנכרן בין ערוצי מידע (קול מול פנים) בשבריר שנייה?
הודעת מעבר למסלול 2 (אם שאלה 3 ≥ 4): "הופס, עצרתי רגע! שמתי לב שסימנת שהסבר מילולי ושקיפות הם קריטיים עבורכם! בארכיטקטורה מהירה וזולה, מנהלים בדרך כלל לא מקבלים הסברים כאלו... אני מנתב אותנו למסלול הניהולי והאמון."

מסלול 2 — 7 שאלות (דיוק ואמון)

  • 1עד כמה חשוב לכם לקבל הסבר ברור במילים (ולא רק ציון יבש באחוזים)?
  • 2עד כמה קריטי למנהלים שלכם להבין איך המערכת הגיעה למסקנה, ולא מ'קופסה שחורה'?
  • 3עד כמה מחמיר ומוגבל התקציב שלכם לרכישת שרתים יקרים וכרטיסי מסך (GPUs)?
  • 4עד כמה השאיפה לאחוזי דיוק מקסימליים (אפילו בתנאי מעבדה) היא הערך החשוב ביותר?
  • 5האם תהיו מוכנים 'לשלם' בזמן עיבוד ארוך יותר, ובלבד שלא נפספס אף ניסיון מרמה?
  • 6עד כמה חשוב לכם שהמערכת תתבסס על קוד פתוח, כדי שלא תהיו תלויים בחברות ענק חיצוניות?
  • 7עד כמה קריטי שהסרטונים וההקלטות הרגישות יעובדו רק בתוך שרתי החברה שלכם?
הודעת מעבר למסלול 1 (אם שאלה 3 ≥ 4): "חכה, בוא נעשה חישוב מסלול מחדש! רציתם מודלים מדויקים ועמוקים, אבל עכשיו סימנת מגבלת תקציב... אני מעביר אותנו לנתיב היעילות והחיסכון."

תוצאות הסיווג ובונוסים

Late Fusion (מיזוג מאוחר): כאשר Operational Efficiency > Accuracy & Trust — מהיר, זול, עמיד לרעשים, מגיב בשבריר שנייה.
Deep Feature Fusion (מיזוג עמוק): כאשר Accuracy & Trust ≥ Operational — דיוק מקסימלי, שקיפות מלאה (XAI), הסברים מילוליים למנהלים.
בונוס — Knowledge Distillation: כאשר ציון כלכלי ≥ 4. שילוב מודל מורה גדול + סטודנט קטן לחיסכון בחומרה.
בונוס — Open-Source LLM: כאשר ציון אסטרטגי ≥ 4. שימוש במודלים כמו Llama 3 לאוטונומיה ופרטיות מלאה.

פרומפטים לבניה עם AI (Copilot Instructions)

  • תמיד התחל כל תגובה עם "Hopa!"
  • כתוב קובץ תכנית (plan file) לפני כל פיתרון — שמור ב-plan/steps/
  • שאל שאלות בתוך קובץ התכנית וחכה לאישור לפני שממשיכים
  • CSS — השתמש ב-skill cssLayer (שכבות נפרדות לכל component)
  • Services — השתמש ב-skill serviceLayer
  • סגנון קוד: אין semicolons, גרש בודד ב-JS, גרשיים כפולים ב-HTML
  • שמות event handlers: onSomething
  • פונקציות רגילות (לא arrows) בייצוא: export function foo() {}
  • פרמטר event: ev (לא e או event)
